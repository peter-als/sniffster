sniffster

A passive network inventory sniffer that watches live traffic, extracts compact metadata, and logs findings for later inventory analysis without trying to become a full IDS.

The current implementation is built around XDP and libbpf, with an intentionally simple, low-overhead runtime. Similar packet observations are coalesced early so downstream code sees less repetition and the tool stays cheap enough to run on a normal development box.

What it does

sniffster observes live traffic, extracts just the metadata it cares about, and turns that into an inventory-oriented stream of logged events. The goal is not deep packet analytics or full intrusion detection, and it is not to assemble the final network picture online. The goal is to capture useful findings efficiently so that the observed subnet can be analyzed offline afterward.

That is why the runtime coalesces similar packet observations early. This is not only a performance trick. For inventory-style sniffing, it is often more useful to reduce repetition near the capture path than to move a large volume of near-duplicate events through the rest of the system.

Design direction

  • Low-overhead capture path The project is built around XDP, libbpf perf buffers, and compact packet metadata.
  • Explicit runtime contracts Queue safety comes from deterministic startup and shutdown ordering, not shared ownership machinery.
  • Current focus Internal API cleanup, profiling, and broader end-to-end validation are the main next steps.

Why I built it

This started as an independent implementation effort after some adjacent network-observability work left me thinking that the same kind of tool could be made leaner and faster.

Part of the fun here is that it pulls me back toward the sort of performance-minded engineering I used to enjoy in HFT: explicit ownership, tight layouts, predictable runtime behavior, and as little unnecessary machinery in the hot path as possible.

It is also why the current version leans so hard into early coalescing and a cheap runtime footprint. The goal is not just correctness; it is to keep the tool light enough that it can sit on a normal development box without becoming its own problem.

Tech

  • C++26 • named modules Core source is organized under network/, sniffer/, and processors/.
  • XDP and libbpf Packet observation and userspace event delivery.
  • CMake, Ninja, Boost, CLI11 Build system, queueing utilities, and command-line plumbing.
  • Linux-first runtime The current implementation targets a Linux environment with modern compiler support.

Current status

This is an actively developed implementation. The XDP/libbpf path is already in place, the runtime is covered by a growing automated test suite, and the design direction is intentionally performance-minded.

Future work includes deeper profiling, more polish around internal APIs, and a possible DPDK-based path for SPAN or mirrored traffic.

Personal notes

  • Allocation discipline The project intentionally avoids heap-managed ownership in the hot path and prefers explicit lifetime rules where that keeps the runtime simpler.
  • Test coverage The automated tests cover core runtime coordination, malformed input handling, coalescing behavior, and queue overflow scenarios, with room to broaden end-to-end validation further.
  • Future direction A DPDK-based path for SPAN or mirrored traffic is one of the obvious experiments once the current XDP path has been profiled more thoroughly.
